# 🗄️ Chapter 5: Database Design

This chapter describes the database architecture, collections, attributes, relationships, validation mechanisms, and sample records used in the **Dhaka Bus Route & Fare Finder** system.

---

## 📚 5.1 Database Overview

The project uses **Cloud Firestore**, a NoSQL cloud database provided by **Firebase**.

Unlike traditional relational databases, Firestore stores data in **collections** and **documents**, making it suitable for hierarchical and flexible data structures.

### 🔑 Key Features

- ☁️ Cloud-hosted NoSQL database
- ⚡ Real-time synchronization
- 📈 Highly scalable
- 🔒 Secure through Firebase Security Rules
- 📱 Seamless integration with Flutter

---

## 📂 5.2 Collections

The database consists of two primary collections.

### 📋 Table 5.1: Database Collections

| Collection | Purpose |
|:-----------|:--------|
| `buses` | Stores all bus information including routes, fares, and map coordinates. |
| `locations` | Stores unique bus stop names for dropdown suggestions and searching. |

---

## 📝 5.3 Attributes, Primary Keys, and Relationships

### 🚌 5.3.1 `buses` Collection

#### 🔑 Primary Key

- **Document ID**
  - Auto-generated by Cloud Firestore.
  - Acts as the primary key.

#### 📋 Fields

| Field | Data Type | Description |
|:------|:----------|:------------|
| `busId` | String | Unique bus identifier (e.g., **DE-001**). |
| `busName` | String | Display name of the bus (e.g., **Dhaka Express**). |
| `stations` | Array<String> | Ordered list of all bus stop names. Used for searching. |
| `faresFromSource` | Map<String, Double> | Stores fare from the first stop to every other stop. |
| `route` | Array<Map> | Complete route containing stop names, GPS coordinates, and fare information. |

#### 📍 Route Object Structure

Each object inside the **route** array contains:

| Field | Data Type | Description |
|:------|:----------|:------------|
| `name` | String | Bus stop name |
| `latitude` | Double | Latitude coordinate |
| `longitude` | Double | Longitude coordinate |
| `fare` | Double | Fare from the first stop |

---

### 📍 5.3.2 `locations` Collection

#### 🔑 Primary Key

- **Document ID**
  - Auto-generated by Cloud Firestore.

#### 📋 Fields

| Field | Data Type | Description |
|:------|:----------|:------------|
| `locationId` | String | Unique identifier generated from timestamp or custom value |
| `locationName` | String | Unique bus stop/location name |

---

### 🔗 5.3.3 Relationships

There is **no direct relational (JOIN) relationship** between the two collections.

Instead:

- The **stations** array references location names.
- The **route** array references the same stop names with GPS coordinates.
- The `locations` collection provides search suggestions.
- The `addLocation()` method prevents duplicate location names.

### 📊 Relationship Summary

| Collection | References |
|:-----------|:-----------|
| `buses` | References stop names stored in `locations` |
| `locations` | Independent collection used for search and dropdowns |

---

## ✅ 5.4 Data Validation

The application validates data at multiple levels.

### 💻 Client-Side Validation (Flutter)

Validation is performed using Flutter widgets.

Features include:

- Required field validation
- `TextFormField` validators
- Numeric fare validation
- Minimum two route stops required
- Duplicate stop prevention

Screens implementing validation:

- `AddBusScreen`
- `EditBusScreen`
- `MapRoutePicker`

---

### 🔒 Server-Side Validation (Firestore)

Firestore Security Rules can enforce:

- Required fields
- Correct data types
- Authentication for Admin operations
- Read/Write permissions

This ensures:

- Data consistency
- Secure access
- Protection against unauthorized updates
